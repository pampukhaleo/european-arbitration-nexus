

# Аудит сайта chea-taic.be

## Резюме

Проведён аудит по направлениям: безопасность, конфигурация, SEO, архитектура, производительность. Найдено **2 критических**, **9 средних** и несколько мелких проблем. Ниже — приоритизированный план исправлений.

---

## 🔴 Критические проблемы (требуют немедленного исправления)

### 1. Privilege escalation на таблице `users`
RLS-политика "Allow users to update their own data" позволяет любому аутентифицированному пользователю обновить собственную строку **без ограничений по колонкам**. То есть пользователь может выставить себе `roles = ['Admin']` или `is_approved = true` и получить админ-права.

**Что делать:** Переписать политику с `WITH CHECK`, запрещающим менять `roles`, `is_approved`, `share_permission`. Защищённые поля должны меняться только админом или через `SECURITY DEFINER` функцию.

### 2. Guild leaders могут менять `roles` пользователей в своей гильдии
UPDATE-политика "Allow guild leaders to approve users in their guild" не имеет `WITH CHECK` — лидер гильдии может менять любые поля строк (включая `roles`) у пользователей своей гильдии.

**Что делать:** Добавить `WITH CHECK`, ограничивающий обновляемые поля только `is_approved` (или иными разрешёнными).

## 🟠 Важные проблемы безопасности

### 3. Утечка `owner_id` в публичной таблице `paintings`
Политика "Public can view published paintings" отдаёт колонку `owner_id` (UUID пользователей) анонимам — позволяет перечислять пользователей.
**Решение:** Создать публичный VIEW `public_paintings` без `owner_id` и переключить публичное чтение на него; либо реализовать column-level security.

### 4. Любой может писать в `access_logs`
Политика на INSERT с `WITH CHECK (true)` позволяет анонимам спамить логи.
**Решение:** Ограничить INSERT аутентифицированными пользователями или перевести запись логов на SECURITY DEFINER функцию.

### 5. Публичный storage bucket разрешает листинг
Любой может получить список всех файлов бакета.
**Решение:** Сузить SELECT-политику на `storage.objects` — отдавать только конкретные объекты по имени, без листинга.

### 6. Хардкод Supabase URL/anon key в `client.ts`
`src/integrations/supabase/client.ts` содержит ключи в коде вместо `import.meta.env.VITE_*`. Anon-ключ не секрет, но хардкод мешает ротации и работе в разных окружениях.
**Решение:** Перевести на переменные окружения из `.env`.

### 7. RLS политика с `USING (true)` на UPDATE/DELETE/INSERT
Линтер нашёл одну такую политику. Нужно ревью и сужение условий.

### 8. SQL-функции без `SET search_path`
3 функции уязвимы к search_path injection (если злоумышленник создаст одноимённую функцию в другой схеме).
**Решение:** Добавить `SET search_path = public` ко всем `SECURITY DEFINER` и обычным функциям.

### 9. Auth конфигурация
- OTP expiry превышает рекомендуемый порог → сократить в Supabase Auth settings.
- Leaked Password Protection отключён → включить в Supabase Auth.
- Postgres устарел и имеет security patches → запланировать апгрейд.

## 🟡 SEO и метаданные

### 10. Канонические URL построены под GitHub Pages
В `Seo.tsx`:
```ts
const canonicalUrl = `${baseUrl}/?${location.pathname.replace(/\/$/, '') || '/'}`;
```
Это формат spa-github-pages с `?/path`. После переезда на нормальный хостинг (Lovable / собственный сервер) canonical должен быть просто `${baseUrl}${location.pathname}` — иначе Google индексирует кривые URL и теряет ссылочный вес.

### 11. `index.html` всё ещё содержит spa-github-pages редирект-скрипт
После ухода с GitHub Pages этот скрипт можно удалить.

### 12. `package.json` ориентирован на gh-pages
`"homepage": "https://chea-taic.be"`, скрипты `predeploy`/`deploy` через `gh-pages`. Если вы публикуетесь через Lovable — эти скрипты больше не нужны и вводят в заблуждение.

### 13. Favicon
`<link rel="icon" type="image/png" href="/favicon.ico" />` — type не совпадает с расширением. Использовать `.png` файл или поменять type на `image/x-icon`.

## 🟡 Архитектура и DX

### 14. Все маршруты импортируются эагерно
В `App.tsx` импортируется ~30 страниц синхронно — увеличивает стартовый бандл. Перевести в `React.lazy()` + `<Suspense>` (особенно для админских / редко используемых страниц: AdminDashboard, GalleryManage, PaintingForm, TokenManagement, QrCodeGenerator).

### 15. `react-helmet` вместо `react-helmet-async`
`react-helmet` не поддерживается, может давать предупреждения и проблемы с SSR. Рекомендуется `react-helmet-async`.

### 16. `manualChunks` в vite ссылается на UI-алиасы
```ts
ui: ["@/components/ui/button", "@/components/ui/card"]
```
Это не сработает корректно — `manualChunks` ожидает имена npm-пакетов, а не локальные модули. Чанк `ui` фактически бесполезен.

### 17. Console.log в production
`src/pages/gallery/TokenManagement.tsx` оставляет `console.log` отладки. Удалить или обернуть в проверку `import.meta.env.DEV`.

### 18. Отсутствует ErrorBoundary
В `App.tsx` нет глобального ErrorBoundary — любая ошибка рендера крашит всё приложение в белый экран.

## 🟢 Мелочи / nice-to-have

- `convert-to-webp.js` — оставшийся скрипт оптимизации изображений в корне; подтвердить, что он используется в build pipeline или удалить.
- `public/sitemap.xml` и `public/rss.xml` существуют — проверить, что они актуальны и попадают в build.
- В `Seo.tsx` нет `hreflang` тегов для мультиязычности (en/fr/ru), хотя сайт трёхязычный — это упускает SEO-возможность.
- `index.html` имеет фиксированный `lang="en"`, не меняется по выбранному языку (Helmet ставит, но при первой загрузке SSR-нет, поэтому индексаторам видно "en").
